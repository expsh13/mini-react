# まえがき

## この本は何のために書いたか

React を使って数年が経ち、`useState` も `useEffect` も当然のように使えるようになった。でも、こんな疑問が頭をよぎったことはないだろうか。

- `useState` はなぜ条件分岐の中で使ってはいけないのか？
- 同じ値を `setState` したとき、React はどのように再レンダリングを省略するのか？（本書では簡略化のため省略するが、仕組みは第7章コラムで解説する）
- `useEffect` のクリーンアップはいつ実行されるのか？
- React はなぜ大量の更新を一括で処理できるのか？

これらの問いに「ドキュメントにそう書いてあるから」以上の答えができるようになるために、この本を書いた。

答えはシンプルだ。**自分でReactを作れば、すべてわかる。**

## この本が他と違う点

「React を自作する」チュートリアルは世の中にたくさんある。しかしその多くは、技術的な不正確さを抱えている。

- "Virtual DOM" という用語を使う（React公式は推奨しない）
- `effectTag` という廃止されたフィールド名を使う
- フックを配列で管理する（実際はリンクリスト）
- `requestIdleCallback` を本番Reactのスケジューリング方式として説明する（本書では教育用の近似として使用し、本番Reactが `MessageChannel` + `shouldYield()` を使うことを明記する）

本書は違う。React 18 のソースコードを実際に参照し、**本番Reactと同じ用語・同じ設計パターン**で実装する。読み終えた後には、`react-reconciler` のソースコードが読める力がついているはずだ。

## 読了後の変化

この本を読み終えると、こう言えるようになる。

> 「`useState` が条件分岐内で使えないのは、Hooksがリンクリストで管理されていて、呼び出し順が変わると対応がズレるからです」

> 「`useEffect` のクリーンアップが次のエフェクト実行前に呼ばれるのは、commit phase の Passive Effects フェーズでまず全クリーンアップを実行してから、新しいエフェクトを実行する設計になっているからです」

**実装した人間だけが持てる、確信を持った説明ができるようになる。**

## 対象読者

- React・TypeScript を実際のプロジェクトで使っている
- React の内部実装に興味があるが、ソースコードを読むのは難しく感じる
- 「なぜそうなるのか」を理解したい

## 用語について

本書では以下の用語を使用する。

| 使用する用語 | 使わない表現 | 理由 |
|---|---|---|
| React elements | Virtual DOM | React公式が推奨しない |
| render phase / commit phase | レンダリングフェーズ | React公式の2フェーズ分類に合わせる |
| `flags`（ビットフラグ） | `effectTag` | React 17以降の正式名称 |
| `return`（Fiberフィールド） | `parent` | コールスタックの「return先」というセマンティクスを保持 |
| `createRoot()` | `ReactDOM.render()` | 本書はReact 18ベース |
